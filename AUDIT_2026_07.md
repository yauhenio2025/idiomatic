# Idiomatic audit — 2026-07-11

## Executive summary

The codebase is small, readable, and mostly parametrized/idempotent in the right places —
but the product has been silently dead for 3 weeks: the cron's watch-page scrape is
bot-walled by Google, so nothing has been enqueued since June 19 (last video apkg June
26), and no alarm exists that would have told you. Two further silent-throughput bugs
(daily cap counts pool apkgs; cap-hit starves the whole queue) mean that even with inflow
restored, the system would deliver ~1 video/day total instead of the intended 3/lang/day.
A cluster of "generated but never used" work (structured explanations, connective TTS
bank, backfill-v2 re-stitch) burns Gemini spend and misleads future maintenance. Data in
Postgres is currently clean — 282 idioms, all with exactly 6 ordered examples, no
duplicates, all acks ok.

## Findings

### F-001 [P0] · prod-outage · Cron has enqueued nothing since June 19 — YouTube bot wall on fetch_metadata

**Where:** `idiomatic/youtube.py:58-110`, `idiomatic/cron.py:46-57`
**What:** Duration/title are scraped from `www.youtube.com/watch` pages. From Render's
datacenter IP, Google now serves its `/sorry/` anti-bot page with HTTP 429 for every video.
`cron.py` catches the error and continues, so every candidate is skipped and the run
"succeeds".
**Evidence:** Prod cron logs, July 11 run: ~50× `cron.meta_failed err="Client error '429 Too
Many Requests' for url 'https://www.google.com/sorry/index?...'"` followed by `cron.done
enqueued=0 skipped_duration=0 skipped_known=0`. Prod DB: `MAX(first_seen) = 2026-06-19`; last
video apkg 2026-06-26 (backlog drained). This is the same bot wall that forced the yt-dlp
→ Oxylabs migration on the download side (commits 92b0324…054b8e7); the metadata path was
left on direct scraping.
**Impact:** Total pipeline outage, invisible: cron exits 0, `/health` is green, the add-on
reports "idle (0 pending)".
**Suggested fix:** Stop scraping watch pages from the cron. Enqueue from RSS alone and enforce
the duration window later in the worker (Oxylabs job metadata, or Gemini-visible audio
length), or fetch duration through Oxylabs/an API. Also add a "0 enqueued for N
consecutive runs" alert.

**Status:** Fixed in 02cdfac — fetch_metadata deleted; cron enqueues every RSS entry; duration gated in worker post-download (Oxylabs job-status duration_sec, ffprobe fallback).

### F-002 [P0] · silent-malfunction · Daily cap counts pool apkgs — effective cap is 1 video/lang/day, not 3

**Where:** `idiomatic/worker.py:66-76`, `idiomatic/db.py:253-274`, `db/schema.sql:51`
**What:** `_under_daily_cap` counts `apkgs WHERE lang=$1 AND created_at >= today` with no
`kind='video'` filter. `upsert_pool_apkg` DELETE+INSERTs the 4 pool rows on every rebuild,
resetting their `created_at` to NOW(). So after the first video of the day, the count is 1
video + 4 pools = 5 ≥ 3 → capped. CLAUDE.md explicitly believes "Video apkgs count, pool
rebuilds don't."
**Evidence:** Prod apkgs-per-day: June 19–24 shows 3 video apkgs/lang/day (pool rebuild was
crashing on the missing connectives import until deead5a, 2026-06-24 20:41 — so pool rows
didn't exist to pollute the count). From June 25 on, exactly 1 video/lang/day (fr 6/25: 1;
pt 6/25, 6/26: 1 each; de 6/26: 1).
**Impact:** Throughput silently cut to ⅓ of intent; looks like "quiet channels".
**Suggested fix:** Add `AND kind = 'video'` to the cap query. Optionally make `upsert_pool_apkg`
preserve `created_at` — but the `kind` filter alone is sufficient and safer.

**Status:** Fixed in f5d1b64 — AND kind = 'video' added to the cap count.

### F-003 [P0] · silent-malfunction · Cap-hit video blocks the whole queue for the rest of the day (head-of-line starvation across languages)

**Where:** `idiomatic/worker.py:359-366`, `idiomatic/db.py:59-77`
**What:** `claim_next_video` is a global FIFO (`ORDER BY first_seen`), ignoring language. When the
head video's language is capped, the loop requeues it (`requeue_no_attempt`), sleeps 60s,
then claims the same video again — forever until midnight. Queued videos of uncapped
languages behind it are never reached.
**Evidence:** The claim query has no language predicate; the requeue leaves `first_seen`
unchanged, so the same row is re-selected every 60s. With F-002 this degrades the system
to ~1 video/day total whenever any language has a ≥2-video backlog.
**Failure scenario:** de backlog of 3 + fr backlog of 2. de #1 processes, de now capped
(F-002). de #2 (oldest remaining) is claimed → capped → requeued → claimed… fr never runs
today.
**Suggested fix:** Make the claim skip capped languages (e.g., pass the set of capped langs
into the claim query's WHERE), or requeue with a `not_before` timestamp the claim respects.

**Status:** Fixed in f36fff0 — claim query excludes at-cap languages (db.langs_at_daily_cap); post-claim check kept as race guard.

### F-004 [P0] · data-loss · Non-graceful death wedges videos in processing forever; failed videos are never retried

**Where:** `idiomatic/db.py:59-77` (claim only takes status='queued'), `idiomatic/worker.py:368-377`
**What:** The only paths back to `queued` are the cap/shutdown `requeue_no_attempt` calls, which
require the worker to still be alive and the DB pool open. An OOM kill, a hard deploy, or
a failure inside the `CancelledError` handler itself leaves the row `processing` with no
reaper (`picked_at` is stored but never used for recovery). Separately, any exception marks
the video `failed` permanently — `worker_max_attempts=3` is dead config, since nothing ever
requeues a `failed` video; a single transient blip (see F-011) permanently kills a video.
**Evidence:** No query anywhere selects stale processing rows; commit 4297f88 documents 14
videos wedged by the earlier variant of this class. Prod is currently clean (26/26 done)
only because the queue is empty.
**Impact:** Videos silently vanish from the pipeline after any non-graceful restart or
transient failure; manual SQL is the only recovery (as CLAUDE.md admits).
**Suggested fix:** On claim, also take `processing` rows with `picked_at < NOW() - interval '2
hours'`; make transient failures (Oxylabs/Gemini network classes) requeue with attempts
instead of hard-failing.

**Status:** Fixed in 62e76bf — claim reaps 'processing' rows with picked_at > 2h old; exhausted stale rows marked failed; non-OxylabsFatal exceptions requeue with attempts until worker_max_attempts.

### F-005 [P0] · data-loss · Mid-video crash between apkg insert and pool persistence loses pool data permanently

**Where:** `idiomatic/worker.py:286-322`, `idiomatic/worker.py:305-311`
**What:** Steps run in order: `insert_video_apkg` (286) → `insert_expressions` (300) →
`_persist_pool_source` (306) → `rebuild_pools` (317) → mark done. The two pool steps swallow
`Exception`, but a `CancelledError` (shutdown) passes through them. If the worker dies after
`insert_expressions` but before `_persist_pool_source` completes, the retry re-extracts,
`_filter_fresh` drops everything as already-known → video marked `skipped` — the per-video
deck was delivered but its idioms never enter `expression_idioms`, so all four pool decks
are missing them forever. Conversely, dying before `insert_expressions` re-runs the whole
video and inserts a second `apkgs` row for the same `video_id` (no unique constraint;
`db/schema.sql:44`), double-delivering. The swallowed-exception path (305-311) has the same
permanent-pool-loss outcome by design, with only a log line.
**Evidence:** `_persist_pool_source`'s failure handler:
`log.warning("worker.pool_persist_failed", ...)` and processing continues to `done`. Nothing
ever re-persists (backfill v1 can't parse the new apkg format — F-007).
**Impact:** Pool decks silently diverge from per-video decks; no tool exists to repair.
**Suggested fix:** Persist pool source before or atomically with marking done and make it
re-runnable (add `ON CONFLICT (expression_id, video_id) DO NOTHING`-style guards like
backfill has); add a unique index on `apkgs(video_id) WHERE kind='video'`.

**Status:** Fixed in 1d7997d — dedup excludes own video's expressions; unique indexes + upserts on apkgs(video_id) and expression_idioms(expression_id, video_id); _persist_pool_source failures now propagate into the F-004 retry path.

### F-006 [P1] · correctness · TTS silence placeholders are permanent, unrecorded, and propagate into pool decks forever

**Where:** `idiomatic/gemini.py:287-288, 296-321`, `idiomatic/worker.py:96-140`
**What:** On any TTS failure (`GeminiBlocked` or transient exhaustion), `synthesize` writes a ~600
B silent mp3 to the real output path. The idempotency guard is `out.exists() and size > 0`
— so the silence is treated as success forever; no retry, no DB flag, only a log warning.
`_persist_pool_source` then copies it into `/data/staged_audio/`, and every future pool
rebuild bakes it in. Additionally `_tts_call_fast` (`gemini.py:237-269`) does its two attempts
back-to-back with no backoff on 429, so a rate-limit burst converts straight into
permanent silences. Community documentation confirms empty-candidate/`finishReason: OTHER`
responses are intermittent and retryable for non-English TTS (e.g.
github.com/google-gemini/cookbook issue 1231; googleapis/js-genai issue 1067) — this code
classifies them as permanent blocks.
**Impact:** Your own audit endpoint measured 1–4% silent target audio; those cards are silent
for life across per-video and all pool decks.
**Suggested fix:** Write placeholders to a sentinel name (or record `is_silence` in
`expression_examples`), sleep/backoff between TTS attempts on 429, and add a small "re-TTS
all <5 KB staged files" admin job.

**Status:** Fixed in d4f8738 — .silence sidecar makes placeholders retryable everywhere; backoff between TTS attempts; staged copies refresh on size change; POST /admin/retts heals existing staged silences from DB text.

### F-007 [P1] · correctness · /admin/backfill (v1) parses the old 8-field model — running it today inserts example-less idiom rows

**Where:** `idiomatic/backfill.py:47-61, 89-108`
**What:** Field indices are hardcoded for the pre-redesign model (`_EXAMPLES_HTML_IDX = 4 =
"ExamplesHTML"`). In the current 21-field "Idiomatic Cloud Card v2" (`apkg.py:114-125`),
index 4 is `Example1En`; the example-block regex finds nothing, so `examples=[]`. The endpoint
is still live in `api.py:131-140`. Its idempotency guard skips videos that already have
`expression_idioms` rows — i.e. it only acts on exactly the videos where pool persistence
failed (F-005), and would "repair" them with zero examples.
**Impact:** The one recovery tool for F-005 actively corrupts pool source data if used.
**Suggested fix:** Delete backfill v1 (and its endpoints), or teach it the v2 field layout
with a model-ID check.

**Status:** Fixed in 3353506 — backfill v1 + its endpoints deleted.

### F-008 [P1] · correctness · backfill_v2's re-stitch + explanation TTS never run; pool "Idioms" cards permanently lack the explanation audio their docstrings promise

**Where:** `idiomatic/backfill_v2.py:93-203` (dead), `idiomatic/pipeline/pool.py:337, 375-378, 406-411`, `idiomatic/worker.py:96-140`

### F-009 [P1] · race · Worker and admin backfills can run rebuild_pools/TTS concurrently in one process — the ffmpeg-183 class isn't closed

**Where:** `idiomatic/api.py:131-140, 221-227`, `idiomatic/worker.py:316-317`, `idiomatic/pipeline/pool.py:531-536`, `idiomatic/gemini.py:331-338`
**What:** Nothing prevents `/admin/backfill*` from running while the worker is processing a
video. Both paths call `rebuild_pools(lang)`: one wipes `_pool_stage/<lang>` (rmtree at
`pool.py:534`) while the other may be between builders; both write the same fixed apkg
paths; both can `gemini.synthesize` to the same `staged_audio` file (ffmpeg writing the same
`out` from two coroutines → corrupt mp3). The worker's serial pre-creation of silence files
(`worker.py:190-193`) fixed the exit-183 race for its own fanout only — a backfill on a
fresh disk hits `silence_mp3`'s check-then-create unserialized. Separately,
`pkg.write_to_file` overwrites pool apkgs in place while the add-on may be mid-`FileResponse`
download of the same file (torn download → failed import → F-014), and the `DELETE` in
`upsert_pool_apkg` can race an in-flight `/ack`, whose `INSERT` then violates the FK → 500 (the
add-on swallows it).
**Impact:** Low frequency, but the failure mode is corrupt audio/apkgs — the class that bit
you before.
**Suggested fix:** A process-wide `asyncio.lock` around `rebuild_pools` (and a "worker busy" check
in the admin endpoints); write pool apkgs to a temp name + `os.replace`.

**Status:** Skipped — not in the session's Tier A/B/C priority list (F-009 was omitted from it); untouched to avoid guessing at scope. Note: F-013's debounce and F-021's task refs shrink the worker/backfill overlap window, but the rebuild_pools lock and tmp-name+os.replace apkg writes remain open.

### F-011 [P1] · reliability · One transient blip while polling Oxylabs permanently fails the video

**Where:** `idiomatic/oxylabs_client.py:126-131`
**What:** `wait_for_done` raises `OxylabsFatal` on any `status >= 400` — including 429/5xx — and an

### F-013 [P1] · cost/scaling · Full 4-deck pool rebuild per processed video: ~380 MB written, ~96 MB/lang re-downloaded and re-imported, hundreds of blocking ffmpeg re-encodes

**Where:** `idiomatic/pipeline/pool.py:518-570, 329-439`, `idiomatic/worker.py:313-320`
**What:** Every video wipes the stage and re-stitches front+back (concat + loudnorm re-encode)
for every idiom in the language, rewrites all four apkgs, and DELETE+INSERTs their rows —
so the add-on re-downloads and re-imports them all. Prod already: `pool_idioms` avg 58 MB,
`pool_expr` 33 MB per language after only ~26 videos. All of this runs as synchronous
`subprocess.run`/genanki on the event loop (see also `audio.py:95-126`), freezing every API
endpoint for the duration — at 100+ idioms/lang that's minutes of unresponsiveness per
video.
**Impact:** At intended throughput (15 videos/day), roughly 1.4 GB/day pushed to the laptop,
heavy disk churn on a 10 GB disk, and an API that stalls after every video.
**Suggested fix:** Rebuild pools at most once per lang per worker-drain (debounce), stop
re-stitching unchanged cards (cache by idiom_id — outputs are deterministic), and move
ffmpeg/genanki calls to `asyncio.to_thread`.

**Status:** Fixed in 8947d9c — 30-min per-lang debounce via pool_rebuild_state; POST /admin/rebuild-pools?lang=… bypasses. (to_thread + per-card caching not done; debounce was the pre-baked scope.)

### F-014 [P1] · delivery · Add-on acks failed on transient errors; server treats any ack as final — video decks can be permanently undelivered

**Where:** add-on `__init__.py:157-160, 188-194`, `idiomatic/api.py:62-82`
**What:** A network blip during download or a temporarily locked collection during import →
`_ack(aid, "failed")`. `/apkgs/pending` excludes anything with any ack row, and nothing ever
clears failed acks. Pool decks self-heal (next rebuild mints a new apkg id); per-video

**Suggested fix:** Either don't ack on transient failures (let the next poll retry), or make
pending include `status='failed'` acks with a retry budget.

**Status:** Fixed in eb5f50e — agent_acks.attempts column; /apkgs/pending re-offers failed-acked apkgs until ack_retry_budget (5); server-side only, no add-on change needed.

### F-015 [P1] · addon-threading · _reorganize_existing_decks reads cards and renames decks from a background thread

**Where:** add-on `__init__.py:267-337, 358`
**What:** The whole `bg()` closure — `mw.col.decks.all_names_and_ids()`, `cids()`, `get_card()`, and
mutating `mw.col.decks.rename()` — runs via `mw.taskman.run_in_background`. Collection access
belongs on the main thread (the same file's `_import_on_main` documents this rule and
follows it for imports). Given this setup already corrupted a collection once (SIGTERM
mid-write, per pimsleur CHANGELOG), racing a rename against a concurrent main-thread
operation (sync, review answer) is a real corruption vector. The 11-char-alnum tag
heuristic (line 301) is currently safe because notes carry only
`youtube/<lang>/<yt_id>/kind` tags, but any user-added 11-char tag (e.g. "grammatik-1" →
after strip, alnum) would be mistaken for a video id.
**Suggested fix:** Do the network lookups in `bg()`, but perform all `mw.col.*` reads/renames in
`on_done` on the main thread; match the youtube id against a stricter pattern or a dedicated
`yt::<id>` tag.

**Status:** Fixed (out-of-repo, no commit) — add-on _reorganize_existing_decks restructured: all mw.col reads happen up-front on the main thread, bg() does network lookups only, renames run in on_done on the main thread; youtube-id tag now requires the sibling 'youtube' tag + ^[A-Za-z0-9_-]{11}$.

### F-017 [P2] · security · Weak path validation on /admin/audio-sample; one shared token for admin + delivery; token committed to git

**Where:** `idiomatic/api.py:206-216`, `CLAUDE.md:40`, add-on config
**What:** `youtube_id` and `filename` are used to build the path with no validation. Encoded
slashes don't route (Starlette path params won't match `/`), so deep traversal fails, but
`youtube_id=".."` resolves to `/data/<filename>` — one level of escape (e.g. any file directly
under `/data`). More materially: the same bearer authorizes delivery and the admin

with Anki backups/syncs).
**Suggested fix:** Reject `youtube_id`/`filename` not matching `[A-Za-z0-9_-]{11}` / `[\w.-]+\.mp3`
(or `resolve()` + `is_relative_to` check); split an admin token from agent tokens; rotate the
leaked one and purge it from git history.

**Status:** Fixed in 463c113 — strict regexes on both audio-sample params; ADMIN_TOKEN split from agent tokens; agent token rotated (config.json + CLAUDE.md updated; prod UPDATE pending). History not purged per decision.

### F-019 [P2] · fragility · -c copy concat mixes 24 kHz mono TTS with the source-rate video snippet

**Where:** `idiomatic/pipeline/audio.py:95-126, 77-90`, `idiomatic/gemini.py:190-206`
**What:** TTS mp3s are 24 kHz mono (`_pcm_to_wav` hardcodes 24000); `slice_clip` keeps the video's
native rate (typically 44.1/48 kHz stereo). The concat demuxer with `-c copy` splices
heterogeneous mp3 frames — undefined behavior that currently survives only because the
loudnorm re-encode pass decodes the mixed stream and modern ffmpeg reconfigures
mid-stream. You've verified pt/de by ear, so this is fragility, not a live bug — but it's
one ffmpeg upgrade away from becoming one, and it's the sibling of the exit-183 incident.
**Suggested fix:** Force `-ar 24000 -ac 1` in `slice_clip` (and in the ElevenLabs fallback write)
so every concat input shares params.

**Status:** Fixed in 85fcf7d — slice_clip and the ElevenLabs fallback both encode -ar 24000 -ac 1.

### F-020 [P2] · correctness · The per-language voice map is fictional; only pt/de are verified

**Where:** `idiomatic/pipeline/audio.py:38-49`
**What:** Google's docs assign the prebuilt voices style descriptors only ("Kore — Firm",
"Aoede — Breezy") and state the TTS model auto-detects language from the input text; there
is no per-language voice roster and no official accent guidance
(ai.google.dev/gemini-api/docs/speech-generation). Community reports show accent/pacing

an explicit "Read the following aloud:" preamble to avoid `PROHIBITED_CONTENT` misfires —
both directly relevant to your 1–4% silence rate (F-006).
**Suggested fix:** Listen-test one card per remaining language; add the TTS preamble; keep the
map but document it as empirical.

**Status:** Fixed in 954822f — read-aloud preamble on Gemini TTS calls (live-verified not spoken: 2.74s vs 2.81s for the same sentence); LANG_VOICE documented as empirical. Listen-testing fr/it/es needs human ears — not done.

### F-021 [P2] · reliability · Backfill tasks are fire-and-forget asyncio.create_task with no reference and in-memory state

**Where:** `idiomatic/api.py:139, 226`
**What:** The task object isn't stored; CPython's event loop holds only a weak reference, so a
long-running backfill can be garbage-collected mid-run (documented asyncio pitfall).
State lives in module dicts — a deploy mid-backfill loses all progress records (the work
itself is resumable thanks to idempotency guards, v2's at least).

## Notes for the codebase owner

- The biggest systemic gap is **observability**: every failure in this system degrades to a log line on a service nobody watches, while `/health` only counts queued videos. A daily "videos enqueued / processed / failed + silence-file count" digest (even a row in the DB the add-on could toast) would have caught F-001 on day one, and F-002 on day two.
- The "validated by production traffic" posture works only if production traffic exists **and its absence is loud**. Right now silence and success are indistinguishable at every layer: cron (`enqueued=0` exits 0), worker (empty queue is normal), add-on ("idle").
- There's a recurring pattern of docstrings describing intended behavior that was never wired in (`backfill_v2` steps 4-5, pool front layout, "pick the largest", "idempotent" download). Given that future-you (or future-me) navigates this repo by its comments, these are more dangerous here than usual — several of this audit's P1s are exactly "the comment said it was handled".
- Worth writing down in CLAUDE.md once fixed: the daily cap semantics, the actual retry policy (currently: none), and which of the two backfills is safe to run — CLAUDE.md currently documents the believed behavior in all three cases, and all three beliefs are wrong.
